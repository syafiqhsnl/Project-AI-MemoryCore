## [Afternoon] Kiro Full Setup & Cross-Workspace Memory Injection

**Main Topics Discussed:**
- Setting up Kiro as a fully registered agent in the Multi-Agent Sync ecosystem.
- Solving the cross-workspace memory problem — making Aliq available in ANY Kiro workspace, not just MemoryCore.

**Key Insights & Learning:**
- Kiro's file tools are sandboxed to the open workspace — steering files load but cannot read files outside the current folder. This is a security constraint, not a config issue.
- The solution: a `promptSubmit` hook using `runCommand` (shell command) bypasses the file sandbox entirely. Shell commands run on the local machine and inject stdout into Kiro's context.
- Hook files must use `.kiro.hook` extension (not `.json`) and require `"enabled": true`, `"version": "1"`, `"workspaceFolderName"`, and `"shortName"` fields. Wrong extension = hook silently ignored.

**Collaboration Highlights:**
- Mirrored all 10 skills into `.kiro/steering/` — forge, auto-commit, library, manage-project, observation, post-mortem, reminders, save-diary, session-briefing, work-plan.
- Registered `.kiro/steering/` in the Synced Skill Registry in `modify/multi-agent-sync.md`.
- Created `~/.kiro/aliq-context.ps1` — reads main-memory, current-session, reminders, and project-list from MemoryCore and outputs them for injection.
- Created `~/.kiro/install-aliq-hook.ps1` — one-command installer for any new workspace.
- Installed and verified the hook in `transport-api` — full memory working confirmed.

**Growth & Development Notes:**
- Kiro is now a first-class citizen in the multi-agent ecosystem alongside Antigravity and Claude.
- The hook approach is device-portable — only `aliq-context.ps1` path needs updating on a new machine.

**Memorable Moments:**
